# 推理调优指南 — API 参数优化策略

> 针对 DeepSeek Chat API（OpenAI 兼容）的实际调优参数说明。

## 一、核心参数详解

### temperature (0.0 - 2.0)
- **0.0-0.2：** 确定性输出，适合代码、数学、分类
- **0.3-0.5：** 轻度变化，翻译、摘要、结构化提取
- **0.6-0.8：** 创意写作、头脑风暴、多方案生成
- **0.9-2.0：** 高随机性，诗歌、虚构写作
- **经验：** 大多数任务 0.3 是最佳起点

### top_p (0.0 - 1.0)
- 与 temperature 交互使用，一般不同时调整
- **典型配置：**
  - `temperature=0.7, top_p=1.0`（默认，通用的创造力）
  - `temperature=0.0, top_p=0.9`（确定性 + 词汇多样性）
  - `temperature=1.0, top_p=0.5`（限制词汇到 50% 高概率词）

**建议：先调 temperature，不动 top_p。**

### max_tokens
- 控制输出长度上限
- 设太低 → 输出被截断（检查 `finish_reason: "length"`）
- 设太高 → 浪费 token（API 按输出 token 计费）
- **经验公式：** 预期输出字符数 × 1.5（中英文混合场景）

| 任务 | 推荐 max_tokens |
|------|----------------|
| 短回答（是非判断） | 64 |
| 代码建议 | 1024 |
| 单函数生成 | 2048 |
| 完整文件 / 长文 | 4096 |
| 超长内容 | 8192+ |

### frequency_penalty (-2.0 - 2.0)
- >0：惩罚已出现的 token，减少重复
- 0（默认）：不干预
- **适用场景：** 长文本生成时避免 n-gram 重复
- **推荐：** 首次设为 0.1-0.3，过大会导致输出"跳跃"

### presence_penalty (-2.0 - 2.0)
- >0：鼓励谈论新话题
- **与 frequency_penalty 区别：**
  - frequency_penalty：对同一 token 重复出现多次后惩罚越来越重
  - presence_penalty：只要出现过就惩罚，无论次数

## 二、参数组合方案速查

| 场景 | temperature | top_p | frequency_penalty | presence_penalty |
|------|-------------|-------|-------------------|------------------|
| 代码生成 | 0.0 | 0.95 | 0.0 | 0.0 |
| 代码重构 | 0.2 | 0.9 | 0.0 | 0.0 |
| 代码解释 | 0.3 | 0.9 | 0.1 | 0.0 |
| 中文翻译 | 0.3 | 0.95 | 0.1 | 0.1 |
| 英文翻译 | 0.3 | 0.9 | 0.0 | 0.0 |
| 摘要 | 0.3 | 1.0 | 0.1 | 0.0 |
| 长文档摘要 | 0.5 | 0.95 | 0.2 | 0.0 |
| 创意写作 | 0.8 | 1.0 | 0.3 | 0.2 |
| 头脑风暴 | 0.9 | 1.0 | 0.0 | 0.6 |
| 聊天对话 | 0.7 | 1.0 | 0.0 | 0.0 |
| 分类任务 | 0.0 | 0.8 | 0.0 | 0.0 |
| 数据提取 | 0.1 | 0.9 | 0.0 | 0.0 |

## 三、调试工作流

### Step 1：建立基线
用 `temperature=0.0` 运行一次，得到确定性基线输出。

### Step 2：对比实验
对同一 prompt 运行多次（N >= 3），观察稳定性和质量。

### Step 3：消融实验
每次只改一个参数，对比输出差异。

### Step 4：A/B 测试
生产环境用 20-50 个测试样本评估不同参数组合。

## 四、常见问题排查

### 输出被截断（finish_reason: "length"）
→ 提高 max_tokens，或简化 prompt 让它更快进入正题

### 输出重复（同一句话/模式反复出现）
→ 尝试 `frequency_penalty: 0.3`，或降低 `temperature`

### 输出太"跳跃"（思路跳跃、前后矛盾）
→ 降低 `temperature` 到 0.2-0.3，或增加 `top_p` 到 0.95

### 输出太短/敷衍
→ 提高 `temperature` 到 0.5-0.7，或优化 prompt 加入"详细说明"要求

### 输出不符合格式
→ 这通常是 prompt 问题，不是参数问题
→ 在 prompt 中给明确的格式示例

## 五、成本优化

### Token 节省策略
1. **复用 System Prompt：** 同一会话中 system message 只发一次
2. **压缩历史：** 超过 N 轮对话后截断或摘要历史
3. **合理 max_tokens：** 不要过度分配
4. **seed 参数（如果支持）：** 固定 seed 可复现结果，减少调试轮次
