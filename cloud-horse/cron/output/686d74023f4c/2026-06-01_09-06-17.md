## Data Sources

### Primary: Tencent Finance API (qt.gtimg.cn) — ✅ Works from China VPS

```bash
# Fetch real-time ETF data
curl -s "http://qt.gtimg.cn/q=sh513650,sz159941"
```

**Format**: GBK-encoded pipe-delimited text:
```
v_shCODE="51~name~code~current~prev_close~open~volume~...~high~low~amount"
```

Key field positions (0-indexed):
| Field | Index | Description |
|-------|-------|-------------|
| name | 1 | Fund name (GBK encoded) |
| current | 3 | Current price |
| prev_close | 4 | Previous close |
| volume | 6 | Volume (手) |
| high | 33 | Day high |
| low | 34 | Day low |
| amount | 37 | Amount (万, multiply by 10000) |

**Exchange prefix rule**:
- Shanghai-listed (codes starting with `51`, `11`): use `sh` prefix → `sh513650`
- Shenzhen-listed (codes starting with `15`, `16`): use `sz` prefix → `sz159941`

**Pitfall**: Response is GBK encoded raw bytes. Decode with `.decode('gbk')`. The raw bytes from `subprocess.run(..., capture_output=True).stdout` can be decoded directly. If using Python requests/curl and the output is already decoded as UTF-8, you cannot re-encode to latin1 — handle at the binary level.

**Pitfall**: This API only returns data during market hours + ~30min after close. Outside trading hours, prices are stale (last close).

### Historical K-Line Data: Tencent Finance API (ifzq.gtimg.cn) — ✅ Works from China VPS

```bash
# Fetch daily K-line data
curl -sL "http://ifzq.gtimg.cn/appstock/app/fqkline/get?param=sh513650,day,2023-01-01,2026-05-27,1000,qfq"
```

**Response format**: JSON with data under either `day` or `qfqday` key depending on the ETF:

```python
day_data = data.get('data', {}).get(key, {})
days = day_data.get('day') or day_data.get('qfqday') or []
```

Each record: `[date, open, close, high, low, volume]`

**Pitfall**: Different ETFs use different keys in the response. 513650 uses `day`, 513100 uses `qfqday`. Always check both with `or` fallback.

**Pitfall**: The API may return data starting from a later date than requested (e.g., requesting from 2023-01-01 but data starts from 2023-09-27 for some ETFs). Check the actual range returned.

**Limitation**: Max 2,000 records per API call. Use chunked fetching (backward iteration with decreasing `check_start` dates) to get full history. See `references/global-index-database.md` for the chunked-fetching implementation and per-index earliest dates.

## Smart DCA Strategy (均线偏离法 / Moving Average Deviation)

An enhancement over fixed DCA: adjust monthly purchase quantity based on the ETF's position relative to its N-day moving average. Buy more when the market is down, buy less at highs.

### 10-Year Backtest Results (2016-2026, S&P 500 + Nasdaq 100)

| Metric | Fixed DCA | Smart DCA | Difference |
|--------|-----------|-----------|------------|
| Total invested | $60,000 | $47,140 | **-21%** |
| Total return | 134% | 144% | **+10%** |
| Annualized return | 8.98% | 9.42% | +0.44% |
| Max drawdown | -22% | -17% | **-5pp** |

**Key insight**: Smart DCA achieves similar or higher returns with 21% less capital deployed — superior capital efficiency. The advantage compounds in bear markets (2020 COVID, 2022 correction) when it buys more at lower prices.

### Adjustment Rules

Use the 60-day (or 12-month) moving average deviation:

```
Price vs MA      │  Factor │  Action
─────────────────┼─────────┼─────────────
> +12% above MA  │  0.4x   │  Heavy reduce (extreme high)
+8% ~ +12%       │  0.6x   │  Reduce
+4% ~ +8%        │  0.8x   │  Light reduce
-4% ~ +4%        │  1.0x   │  Normal (baseline)
-4% ~ -8%        │  1.2x   │  Light increase
-8% ~ -12%       │  1.5x   │  Increase
< -12% below MA  │  2.0x   │  Heavy increase (extreme low)
```

### Implementation

```python
def smart_dca_amount(base_amount, current_price, ma_price):
    """Return adjusted investment amount based on MA deviation."""
    if ma_price <= 0:
        return base_amount
    dev = (current_price - ma_price) / ma_price * 100
    
    if dev > 12:    return base_amount * 0.4
    elif dev > 8:   return base_amount * 0.6
    elif dev > 4:   return base_amount * 0.8
    elif dev > -4:  return base_amount * 1.0
    elif dev > -8:  return base_amount * 1.2
    elif dev > -12: return base_amount * 1.5
    else:           return base_amount * 2.0
```

For share-based investing (Chinese ETFs trade in lots of 100 shares):
```python
base_shares = 500  # baseline
deviation = +8.4   # current deviation %
if deviation > 8:      adj = int(base_shares * 0.6 / 100) * 100  # 300 shares
elif deviation > 4:    adj = int(base_shares * 0.8 / 100) * 100  # 400 shares
elif deviation > -4:   adj = base_shares                          # 500 shares
elif deviation > -8:   adj = int(base_shares * 1.2 / 100) * 100  # 600 shares
else:                  adj = int(base_shares * 1.5 / 100) * 100  # 700 shares
```

**Pitfall**: Round to nearest 100 shares (Chinese ETF minimum trade unit). The adjusted amount may differ slightly from the ideal factor.

**Pitfall**: In a prolonged bull market, the strategy keeps buying less, accumulating cash. This is by design — the cash reserve gets deployed when the inevitable correction comes. Do NOT abandon the strategy just because you're "missing out" during a rally.

### Smart DCA Recommendation Logic

The `etf_db.py rec` command computes a monthly recommendation using the 60-trading-day MA deviation:

1. Calculate deviation: `(current_price - MA_60) / MA_60 * 100`
2. Apply rule table to get multiplier
3. Multiply baseline shares by multiplier, round to nearest 100

The recommendation is stored in the database under `smart_dca.last_recommendation` and printed in the daily report.

### 🚨 Manual Fallback: When DB is Corrupted

If `etf_db.py rec` (or any DB command) fails with `json.decoder.JSONDecodeError` (truncated/corrupted DB), **do not re-init** — the DB has 40K+ records. Use the following procedure to manually compute the monthly recommendation:

**Step 1: Fetch current price from Tencent real-time API**
```bash
curl -s "http://qt.gtimg.cn/q=sh513650" | iconv -f GBK -t UTF-8
# Field 3 (0-indexed) = current price
```

**Step 2: Fetch 60+ trading days of K-line data**
```bash
curl -sL "http://ifzq.gtimg.cn/appstock/app/fqkline/get?param=sh513650,day,2026-01-01,2026-05-28,100,qfq" | python3 -c "
import json, sys
data = json.load(sys.stdin)
# Try 'day' then 'qfqday'
d = data.get('data', {}).get('sh513650', {})
days = d.get('day') or d.get('qfqday') or []
closes = [float(day[2]) for day in days]
ma_60 = sum(closes[-60:]) / 60
print(f'MA60: {ma_60:.4f}')
"
```

**Step 3: Compute deviation and apply rules**
```python
dev = (current_price - ma_60) / ma_60 * 100
# Apply: >+12→0.4x, +8~+12→0.6x, +4~+8→0.8x, -4~+4→1.0x, -4~-8→1.2x, -8~-12→1.5x, <-12→2.0x
shares = int(500 * factor / 100) * 100  # round to 100
```

### A类 vs C类 for Long-Term定投

**Key insight**: For a 5-10 year定投 horizon, **A类 is ALWAYS cheaper than C类**, even when申购费不打折.

| Scenario | A类 total cost (10yr) | C类 total cost (10yr) | Winner |
|----------|----------------------|----------------------|--------|
| 申购费打1折 (0.12%) | **~88元** (20元/天) | ~1,460元 | **A类 ×17 cheaper** |
| 申购费不打折 (1.2%) | **~876元** (20元/天) | ~1,460元 | **A类 ×1.7 cheaper** |

**Why C类 is a trap for long-term定投:**
- C类 charges 0.4%-0.8%/year 销售服务费 on TOTAL ASSETS
- As the portfolio grows (e.g., ¥70K+ by year 10), the service fee grows with it (~¥292/year in year 10 alone)
- A类's申购费 is a one-time cost per purchase and does NOT compound

**When C类 makes sense:** Only for holding periods <2 years, or when the user might frequently switch funds.

**Pitfall**: Never recommend C类 for a user who has stated they will hold 5-10 years. The service fee compounds and becomes vastly more expensive.

## ETF Buy Decision Tool (ETF Picker)

The `scripts/etf_picker.py` tool helps decide **which specific ETF to buy right now** by comparing all available S&P 500 (or Nasdaq 100) ETFs on real-time premium rates, liquidity, and fees.

### When used

- **Every month before the user's scheduled purchase (20th)** — the daily report automatically includes a comparison table
- **On-demand** — user asks "which one should I buy now?" and the agent runs the script

### Decision Methodology

The tool ranks ETFs by a **composite score** that prioritizes:

| Factor | Weight | Why |
|--------|--------|-----|
| **Premium rate** (溢价) | Highest | Buy the ETF trading at the lowest premium (or a discount) — even 0.3% cheaper on entry compounds |
| **Liquidity** (成交额) | Medium | Filter out ETFs with <¥0.1亿 daily volume (can't execute reliably) |
| **Fee** (费率) | Low | Long-term cost matters but differences are small (0.05-0.20%) |

**Scoring formula:**
```
score = -premium% × 5 + min(daily_volume_yi × 2, 10) - (fee% - 0.6) × 10
```
- Premium: heavily penalized (each 1% premium = -5 points)
- Liquidity: rewards up to 10 points (5亿+ daily = max)
- Fee: small penalty (0.75% vs 0.60% = -1.5 points)

### Data Sources Used

- **Real-time price + volume**: Tencent Finance API (`qt.gtimg.cn`) — GBK-encoded pipe-delimited
- **Fund NAV (净值/估算净值)**: 天天基金 API (`fundgz.1234567.com.cn/js/CODE.js`) — ✅ Works from China VPS with curl, ✅ Works with Python urllib (single requests fine, batch may timeout)

**Working approach (curl):**
```bash
curl -sL "https://fundgz.1234567.com.cn/js/513650.js" -H "User-Agent: Mozilla/5.0"
# Returns: jsonpgz({"fundcode":"513650","name":"标普500ETF南方","jzrq":"2026-05-26","dwjz":"1.8495","gsz":"1.8565","gszzl":"0.38","gztime":"2026-05-28 04:00"});
```

**Parsing in Python:**
```python
import urllib.request, json
req = urllib.request.Request("https://fundgz.1234567.com.cn/js/513650.js",
                             headers={'User-Agent': 'Mozilla/5.0'})
resp = urllib.request.urlopen(req, timeout=10).read().decode('utf-8')
json_str = resp[resp.index('{'):resp.rindex('}')+1]
nav_data = json.loads(json_str)
gsz = float(nav_data.get('gsz', 0))    # 估算净值 (estimated NAV)
dwjz = float(nav_data.get('dwjz', 0))  # 最新净值 (latest official NAV)
jzrq = nav_data.get('jzrq', '')        # 净值日期
gszzl = nav_data.get('gszzl', '0')     # 估算涨幅 %
gztime = nav_data.get('gztime', '')    # 估算时间
```

**Pitfall**: The `gztime` is in China time (UTC+8) and reflects when the US close NAV was last computed — generally 04:00-05:00 CST (after US market close). So when A-share market opens at 09:30 CST, the gsz is already ~5 hours old. This is normal for QDII ETFs.

**Pitfall**: Batch-fetching NAV for all 11 ETFs in a single Python script may timeout (the fundgz API is slower). Either fetch them one-at-a-time with curl in a shell loop, or add generous per-request timeouts (10s+).
- **Premium calculation**: `(market_price / NAV - 1) × 100`
  - Uses estimated NAV (gsz) when available during trading hours
  - Falls back to latest official NAV (dwjz) when gsz unavailable

### Key Rules

1. **Never buy at >1% premium** — wait for intraday pullback or pick a lower-premium alternative
2. **Exclude ETFs with <¥0.1亿 daily volume** — bid/ask spreads too wide
3. **If best choice is low-liquidity**, warn the user about potential slippage
4. **Premium data is only meaningful during A-share trading hours** (9:30-15:00 CST). After hours, premiums appear inflated because US markets moved since the last NAV fix. Always run the comparison during market hours before making a decision.
